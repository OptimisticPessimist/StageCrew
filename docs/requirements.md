# 演劇団体向けタスク管理SaaS - 要件定義書

## Context

演劇団体では、演出部・技術部・制作部など複数部門が並行して動きつつ、「本番日」という絶対的なデッドラインに向かって収束させる必要がある。Backlogのような汎用ツールでは演劇特有のフェーズ管理や部門間連携が表現しにくいため、演劇に特化したタスク管理SaaSを新規に構築する。

pscweb3とは完全に独立したサービスとして開発。数団体〜数十団体が利用する小規模SaaS。フリーミアム or セルフホスト可能なOSS形態も視野。

**対象**: 物理演劇だけでなく、VR演劇（VRChat, cluster等）の制作管理にも対応する。

## 1. ユーザーとロール

### 1.1 組織構造
```
団体 (Organization)
├── 公演 (Production) ← プロジェクト単位
│   ├── 部門 (Department) ← カスタマイズ可能
│   │   ├── 演出部: 脚本家, 演出, 演出助手, 舞台監督
│   │   ├── 技術部: 照明, 音響, 映像, 舞台美術, 衣装, 小道具
│   │   └── 制作部: 制作進行, 広報, 当日運営
│   └── キャスト (Cast) ← タスク割当可能（衣装アバター組込み、自主練等）
```

### 1.2 アクセス制御: メンバーシップ + ケイパビリティ方式

**設計思想:** 1人が「役者 + 照明スタッフ + 演出助手」のように複数の立場を兼任することが演劇では常態。
純粋なRBACではロール爆発が起き、純粋なABACは実装コストが高い。
→ **メンバーシップ（所属）にケイパビリティ（権限）を付与し、加算合成する方式**を採用。

**構造:**
```
User（ユーザー）
├── OrganizationMembership（団体メンバーシップ）
│   └── org_role: owner | admin | member
├── ProductionMembership（公演メンバーシップ）× 複数可
│   ├── DepartmentMembership[] × 複数部門兼任可
│   │   ├── department: 演出部, 技術部, etc.
│   │   ├── staff_role: 演出, 照明, etc.（部門内の役職）
│   │   └── capabilities: [task.create, task.assign, task.manage, budget.view, ...]
│   ├── CastMembership（キャスト所属）※任意
│   │   └── capabilities: 団体/公演設定で定義（閲覧のみ〜フル権限）
│   └── production_role: manager | member
```

**ケイパビリティ一覧（例）:**
| ケイパビリティ | 説明 |
|----------------|------|
| `task.view` | タスクの閲覧 |
| `task.create` | タスクの作成 |
| `task.edit_own` | 自分が担当のタスクを編集 |
| `task.edit_dept` | 所属部門のタスクを編集 |
| `task.edit_all` | 全タスクを編集 |
| `task.assign` | タスクの担当者割当 |
| `task.delete` | タスクの削除 |
| `comment.create` | コメント投稿 |
| `budget.view` | 予算の閲覧 |
| `budget.manage` | 経費登録・予算管理 |
| `member.manage` | メンバー管理 |
| `settings.manage` | 公演設定の管理 |

**権限の解決ルール:**
1. ユーザーの全メンバーシップからケイパビリティを収集
2. **加算合成（Union）**: いずれかのメンバーシップで許可されていれば許可
3. 例: 役者（`task.view, task.edit_own`）+ 照明スタッフ（`task.create, task.edit_dept`）→ 全て許可
4. 部門スコープ付きケイパビリティ（`task.edit_dept`）は所属部門にのみ適用

**プリセット（設定の簡素化）:**
- 団体設定で「キャストのデフォルト権限」をプリセットから選択可能:
  - 閲覧のみ: `task.view, comment.create`
  - 標準: `task.view, task.edit_own, task.create, comment.create`
  - フル: メンバーと同等

### 1.3 部門のカスタマイズ
- 団体ごとに部門を自由に追加・編集・削除可能
- デフォルトテンプレートとして「演出部」「技術部」「制作部」を用意
- 各部門にスタッフロール（役職）を定義可能
- 1ユーザーが複数部門に所属可能（兼任対応）

## 2. コア機能

### 2.1 課題（Issue）管理
Backlogの課題に相当する中心機能。

**課題の属性：**
- タイトル / 説明（Markdown対応）
- 課題種別: タスク / バグ・問題 / 要望 / 連絡事項
- ステータス: 未対応 → 処理中 → 処理済み → 完了（カスタマイズ可能）
- 優先度: 高 / 中 / 低
- 担当者（複数可）
- 所属部門
- 期限日
- 公演フェーズ（企画/稽古/仕込み/本番/撤収 等）
- 親課題・子課題（階層構造 → WBS対応）
- 添付ファイル（衣装デザイン画、音源、図面等）
- コメント（スレッド形式）
- ラベル/タグ（自由設定）

**部門横断の仕組み：**
- 課題に「関連部門」を複数タグ付け可能
- 例: 「衣装デザイン確定」は演出部が起票 → 技術部（衣装）が担当
- 部門間の依存関係を「ブロック/ブロックされている」で表現

### 2.2 カンバンボード
- 部門ごとのカンバンビュー（デフォルト）
- 全体横断のカンバンビュー
- ステータス列のカスタマイズ（部門ごとに異なるワークフロー可）
- ドラッグ&ドロップでステータス変更
- フィルタ: 部門 / 担当者 / フェーズ / 優先度 / ラベル

### 2.3 ガントチャート
- 公演全体のタイムライン表示
- 本番日を基準に逆算表示できる機能
- 部門ごとの色分け
- 依存関係の矢印表示
- マイルストーン表示（顔合わせ、通し稽古、ゲネプロ、初日、千秋楽）
- ドラッグでの期間調整

### 2.4 WBS（作業分解構造）
- 課題の親子関係をツリー表示
- 部門ごとのWBS + 全体WBS
- 進捗率の自動集計（子タスクの完了率から算出）
- ガントチャートとの連動

### 2.5 ダッシュボード
- 公演ごとの進捗サマリー（本番まであとN日）
- 部門別の進捗率
- 自分の担当タスク一覧
- 期限切れ/期限間近のタスク警告
- 最近の更新アクティビティ

## 3. 演劇特有の機能

### 3.1 公演フェーズ管理
```
企画期間 → 脚本・読み合わせ → 稽古期間 → 仕込み → ゲネプロ → 本番期間 → 撤収（バラシ）
```
- フェーズはカスタマイズ可能（追加・名前変更・順序変更）
- フェーズごとにデフォルトタスクテンプレートを設定可能（後述 3.2）
- 現在のフェーズをヘッダーに常時表示
- フェーズ切替時に次フェーズのタスクを一括生成（オプション）

### 3.2 タスクテンプレート（Nice to have）
- 公演フェーズ × 部門 のマトリクスでデフォルトタスクを定義
- 例:
  - 企画期間 × 制作部: 「会場候補リストアップ」「予算案作成」「出演者募集」
  - 稽古期間 × 技術部: 「照明プラン作成」「音響素材収集」「舞台図面作成」
  - 仕込み × 技術部: 「照明仕込み」「音響仕込み」「舞台セット搬入・組立」
- 団体独自のテンプレートを保存・再利用可能

### 3.3 本番カウントダウン
- 本番日（初日）を設定すると「あとN日」を常時表示
- ガントチャート上で本番日をハイライト
- 期限設定時に「本番N日前」という相対指定が可能

### 3.4 予算・経費管理
- 公演単位の予算枠設定
- 部門ごとの予算配分
- 経費の登録（項目名・金額・カテゴリ・レシート添付・承認ステータス）
- カテゴリ例: 会場費 / 衣装費 / 小道具費 / 照明機材 / 音響機材 / 印刷費 / 広告費 / 交通費 / その他
- 予算消化率のダッシュボード表示
- 経費とタスクの紐付け（「照明機材レンタル」タスク → 経費登録）

### 3.5 稽古場・会場管理
- 稽古場・劇場の情報登録（名称、住所、キャパシティ、利用料金、設備情報）
- カレンダーでの予約状況表示
- 稽古スケジュールとの自動連携
- 会場の利用可能時間・注意事項の共有

### 3.6 VR演劇対応
VR演劇特有のワークフローに対応する。

**VRプラットフォーム管理:**
- 対応プラットフォーム設定（VRChat / cluster / Resonite 等）
- プラットフォームごとの制約・仕様メモ
- ワールドアップロード・公開ステータスの管理
- インスタンス管理（本番用・リハ用・テスト用）

**3Dアセット管理:**
- アセット一覧（アバター、ワールド、ギミック、小道具、エフェクト等）
- アセットごとの進捗ステータス（モデリング → テクスチャ → リギング → 実装 → テスト）
- **ライセンス規約チェック機能:**
  - 利用素材ごとにライセンス種別を記録（CC-BY / 購入 / オリジナル / 改変許可等）
  - 商用利用可否・改変可否・クレジット表記要否のチェックリスト
  - ライセンス違反のリスクがある素材に警告表示
  - クレジット一覧の自動生成（パンフレット・概要欄用）
- アセットの担当者割当（モデラー、テクスチャ担当等）
- 参考画像・3Dプレビューの添付

**VR特有の部門テンプレート:**
- デフォルト部門に「ワールド制作部」「アバター制作部」「ギミック・演出部」等を追加可能
- VR公演フェーズ例: 企画 → 脚本 → ワールド制作 → アバター制作 → ギミック実装 → リハーサル → 本番 → アーカイブ

**VR演劇におけるキャストのタスク:**
- VR演劇では役者自身がアバターへの衣装組み込み等の技術作業を行う
- キャスト向けタスク例:
  - 「衣装アバターへの組み込み・動作確認」
  - 「自アバターの表情設定（Expression Menu）」
  - 「ギミック操作のリハーサル」
  - 「ワールド内動線の確認」
- キャストにもタスク割当・ステータス管理がフルに必要（メンバーと同等権限）

## 4. 部門間連携

### 4.1 クロスボード
- 全部門のタスクを横断的に表示するビュー
- 部門フィルタで絞り込み可能

### 4.2 依存関係
- 課題間に「ブロック / ブロックされている」関係を設定
- 依存先が未完了の場合に警告表示
- ガントチャート上で依存線を表示

### 4.3 共有マイルストーン
- 部門を跨ぐマイルストーン（通し稽古、ゲネプロ等）
- マイルストーンに紐づく全部門のタスク進捗を一覧表示

### 4.4 メンション・通知
- コメントで `@ユーザー名` `@部門名` メンション
- 部門リーダーへの自動通知（担当部門のタスク更新時）

## 5. 通知

- **Discord Webhook**: チャンネルごとに部門を分けて通知可能
  - タスク作成/更新/完了時
  - コメント追加時
  - 期限リマインダー（N日前に通知）
  - マイルストーン到達時
- **アプリ内通知**: 未読バッジ付き通知センター
- **メール通知**: ダイジェスト（日次/週次 選択可能）

## 6. 技術構成（案）

pscweb3と同じ技術スタックを採用し、知見を活かす。

| レイヤー | 技術 |
|----------|------|
| Backend | Python 3.12+ / FastAPI (async) |
| Database | PostgreSQL |
| ORM | SQLAlchemy (async) |
| Frontend | React 19 / TypeScript / Vite |
| CSS | Tailwind CSS v4 |
| 状態管理 | TanStack Query |
| 認証 | Discord OAuth 2.0 + メール認証（追加） |

| カンバン | react-beautiful-dnd or dnd-kit |
| インフラ | Azure or セルフホスト対応（Docker） |
| 通知 | Discord Webhook / SendGrid |

## 7. データモデル（主要エンティティ）

```
Organization（団体）
├── members[]
├── Production（公演）
│   ├── production_type: "real" | "vr" | "hybrid"
│   ├── phases[] ── PhaseTemplate
│   ├── milestones[]
│   ├── Department（部門）
│   │   ├── staffRoles[]
│   │   ├── members[]
│   │   └── statusFlow[] (カスタムステータス定義)
│   ├── Issue（課題）
│   │   ├── type, status, priority
│   │   ├── assignees[]
│   │   ├── department
│   │   ├── phase
│   │   ├── parent_issue / children[]
│   │   ├── dependencies[] (blocks/blocked_by)
│   │   ├── labels[]
│   │   ├── attachments[]
│   │   └── comments[]
│   ├── Board（ボード）
│   │   └── columns[] (ステータス列定義)
│   ├── Budget（予算）
│   │   ├── total_amount
│   │   ├── department_allocations[]
│   │   └── Expense（経費）
│   │       ├── amount, category, receipt, status
│   │       └── related_issue
│   ├── Venue（会場・稽古場）
│   │   ├── name, address, capacity, hourly_rate
│   │   ├── facilities[], notes
│   │   └── VenueBooking（予約）
│   └── VRAsset（3Dアセット）※VR公演時
│       ├── asset_type: avatar | world | gimmick | prop | effect
│       ├── status: modeling → texturing → rigging → implementing → testing → done
│       ├── assignees[]
│       ├── platform: vrchat | cluster | resonite
│       └── License（ライセンス）
│           ├── license_type: CC-BY | purchased | original | etc
│           ├── commercial_use, modification_allowed
│           ├── credit_required, credit_text
│           └── warning_flags[]
```

## 10. MVP（v1.0）スコープ

Phase 1で実装する最小限の機能:

1. **認証**: Discord OAuth ログイン
2. **団体・公演管理**: CRUD、メンバー招待
3. **部門管理**: カスタマイズ可能な部門・ロール
4. **課題管理**: 作成・編集・ステータス管理・担当割当・親子関係
5. **カンバンボード**: 部門別 + 全体ビュー
6. **ガントチャート**: タイムライン表示 + 本番カウントダウン
7. **Discord通知**: タスク更新通知
8. **ダッシュボード**: 公演進捗サマリー

Phase 2:
- WBSツリービュー
- タスクテンプレート（フェーズ×部門マトリクス）
- 依存関係の可視化
- 予算・経費管理
- 稽古場・会場管理

Phase 3:
- VR演劇対応（3Dアセット管理、ライセンスチェック、プラットフォーム管理）
- メール認証（Discord以外の認証手段）
- セルフホスト用Docker対応
- 課金機能
- クレジット一覧自動生成

## 11. 検証方法

- ローカル開発環境で動作確認（Docker Compose for PostgreSQL）
- 演劇制作の典型シナリオでウォークスルー:
  1. 団体を作成、公演を作成
  2. 演出部・技術部・制作部を設定し、メンバーを割当
  3. 各部門でタスクを作成、カンバンで管理
  4. ガントチャートで全体スケジュールを確認
  5. Discord通知が飛ぶことを確認
